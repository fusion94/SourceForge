Vorwort

Hallo SourceForge-Neulinge!

Ich habe diese Anleitung geschrieben, um die Einrichtung eines neuen Projektes und die ersten Schritte bei SourceForge zu vereinfachen. Das ganze ist noch unvollständig und wird (hoffentlich) mit SourceForge mitwachsen.

Die folgende Situation dürfte auf die meisten Programmierer zutreffen:
Es ist ein Verzeichnis vorhanden, in dem der geschriebene Code liegt, vielleicht sogar ein aktuelles CVS repository. Die Features von SourceForge sollen genutzt werden, um das Projekt effizient weiterzuentwickeln, der Öffentlichkeit vorzustellen, vielleicht ein paar Helfer zu finden und natürlich auch neue Benutzer. Ich gehe im folgenden daher einfach mal von dieser Situation aus. Also los...

Überblick

Die folgende Liste zeigt, was zum Einrichten eines neuen Projektes bei SourceForge gemacht werden muß. Die Erklärung dazu kommt gleich.

  1. Als neuer User bei SourceForge anmelden
  2. Mit diesem Account einloggen und das neue Projekt anmelden
  3. Gruppen- und Projektinformationen anpassen
  4. CVS repository mit dem aktuellen Schnappschuß des Quellcodes einrichten
  5. Webseiten für das Projekt auf den Webserver laden
  6. Mailinglisten anlegen
  7. ggf. mit CVS vertraut machen und mit der Arbeit loslegen (bzw. weitermachen)!
... außerdem könnte man noch folgendes brauchen:
  1. Ein Filemodul freigeben, damit man sich den Fortschritt des Projekts ansehen kann
  2. SourceForge "Site Documentation" und das Hilfe-Forum durchlesen

An die Arbeit...

1. Als neuer User bei SourceForge anmelden

Dazu muß man auf der SourceForge-Homepage die Option "New User via SSL" anklicken. Damit der neue Account eingerichtet werden kann, werden einige persönliche Daten abgefragt. Hat man das hinter sich gebracht, kann man sich ab jetzt immer ¨ber die Option "Login via SSL" anmelden.

Nach dem Login befindet man sich auf seiner persönlichen Seite. Hier wird beispielsweise aufgelistet, an welchen Projekten man teilnimmt. Am linken Rand dieser Seite findet man einige Optionen, von denen uns vor allem "Register New Project" interessiert.

2. Das neue Projekt anmelden

Nun muß man sich durch eine Reihe Seiten durcharbeiten und die gewünschten Informationen angeben. Das ganze ist eigentlich recht verständlich geschrieben und einfach zu handhaben, daher gehe ich auf die einzelnen Angaben hier nicht weiter ein.

Wenn man alle Angaben gemacht hat, schickt man die Beschreibung des Projekt an das SourceForge-Team. Das Projekt wird dann begutachtet und überprüft, ob es den Zielen von SourceForge entspricht. Keine Sorge, ich habe bis jetzt nicht gehört, daß ein Projekt abgelehnt wurde... wenn es unter einer OpenSource-Lizenz steht und frei verfügbar ist, sollte es klappen.

Nun muß man ein wenig warten. In der Regel bekommt man innerhalb von 24 Stunden eine Mail mit dem Hinweis, daß das Projekt eingerichtet wurde. Wenn man sich nun wieder einloggt, findet man am unteren Ende seiner persönlichen Seite einen Link zum Verwaltungsbereich des neuen Projekts.

3. Gruppen- und Projektinformationen anpassen

Hat man den Link auf seiner persönlichen Seite angeklickt, kommt man also zum Verwaltungsbereich des neuen Projektes. Hier kann man zahlreiche Einstellungen vornehmen.

Bis jetzt hat mal also folgendes:

  • eine Website unter http://Projektname.sourceforge.net
  • ein CVS repository unter cvs.Projektname.sourceforge.net:/cvsroot/Projektname
  • einen anonymous FTP-Zugang unter ftp.Projektname.sourceforge.net
  • und Zugriff auf einen Mailinglisten-Server

Als erstes sollte man eine Beschreibung des neuen Projektes angeben. Dazu klickt man auf "Project Admin", dann auf "Edit Group Public Information and Categorization". Hier werden die gewünschten Informationen in die entsprechenden Felder eingegeben. Die Beschreibung und die Zielplattform(en) werden in der Software Map angezeigt, in der alle Projekte bestimmten Kategorien zugeordnet und beschrieben werden. Ein Link zur Software Map ist am linken Rand oben zu finden.

Die nächste Frage, die interessiert, ist: gibt es weitere Entwickler, die an diesem Projekt mitarbeiten? Wenn ja, dann müssen sie sich alle bei SourceForge wie unter 1. beschrieben anmelden und Ihnen die Usernamen mitteilen. Sie können dann die Namen in die Liste der Entwickler aufnehmen und ihnen Schreibzugriff auf den CVS-Server und die Website geben. Dazu wählt man im Verwaltungsbereich des Projektes "Add Group Member" und gibt die entsprechenden Namen ein.

Der Rest der Einstellungen im Verwaltungsbereich wird später interessant. Nun sollte man erstmal etwas Code bereitstellen...

4. CVS repository mit dem aktuellen Schnappschuß des Quellcodes einrichten

Wenn man mit CVS (wie ich) noch keine Erfahrung hat, braucht man ein bisschen Eingewöhnungszeit. Den Neulingen möchte ich kurz beschreiben, was CVS überhaupt macht:

CVS führt Buch über die Veränderungen, die am Sourcecode vorgenommen wurden, zusammen mit einer Beschreibung dessen, was verändert wurde. Man kann sich zu jeder Zeit jede vorangegangene Version jeder Datei im repository ansehen. Durch CVS können mehrere Entwickler zur gleichen Zeit an der gleichen Datei arbeiten und ihre Änderungen zusammenführen. Wenn zwei Änderungen sich gegenseitig beeinflussen, werden die Entwickler darauf hingewiesen und sichergestellt, daß der Konflikt behoben wird, bevor die neue Version der Datei im repository aufgenommen wird.

Das ist eine schöne Sache, an die man sich zwar erst ein wenig gewöhnen muß, die man aber bald schätzen wird. Im Wesentlichen besteht der Entwicklungszyklus aus:

  1. Man importiert den kompletten Sourcecode ins repository; anschließend checkt jeder Entwickler eine Arbeitskopie aus dem Entwicklungsbaum aus.
  2. Nun arbeitet jeder Entwickler in seiner gewohnten Umgebung am Programm. Sobald ein neues Feature fertig ist, aktualisiert man zunächst die lokale Kopie, um auf dem laufenden zu sein und schreibt die veränderten Files zurück in den Entwicklungsbaum.
  3. Sollten beim zurückschreiben der Veränderungen irgendwelche Konflikte auftauchen, wird man darauf hingewiesen und muß anschließend manuell entsprechende Änderungen vornehmen. In den Dateien werden die veränderten Bereiche mit ">>>>>" hervorgehoben und beide Versionen angezeigt. Hier löscht man entweder einfach die alte Version oder paßt den Bereich an, damit er wie gewünscht arbeitet. Wenn alle Konflikte beseitigt sind, geht man wieder zurück zu Punkt 2 und setzt die Arbeit fort.

Gut. Nun zu den Details, wie man den eigenen Source-Baum auf den CVS-Server von SourceForge lädt. Als erstes sollte man sich die "Site Documentation" zum Thema CVS ansehen. Will man mehr wissen, ist der Griff zum 180 Seiten starken (und frei verfügbaren) CVS-Handbuch angesagt. Dann geht man wie folgt vor:

SSH und CVS für das benutzte Betriebssystem besorgen. Für die Kombination Linux und bash gibt man nun ein:

export CVS_RSH=ssh
export CVSROOT=Username@cvs.Projektname.sourceforge.net:/cvsroot/Projektname

Mit der ersten Zeile sagt man CVS, daß es mit Hilfe von SSH die Verbindung zum repository aufnehmen soll. Das wird aus Sicherheitsgründen so gemacht. Die zweite Zeile teilt CVS mit, wo das repository genau zu finden ist. Die kursiven Teile (Username und Projektname) müssen entsprechend angepaßt werden!

Oft wird man nun den Sourcecode aus einem Verzeichnis-Baum in den CVS-Server importieren wollen. Dazu wechselt man in das Verzeichnis, daß übertragen werden soll, und gibt folgende Zeile ein:

cvs import Verzeichnisname vendor start

Verzeichnisname ist der Name, unter dem das repository angesprochen werden soll. Wenn alles klappt, wird man nun nach seinem SourceForge User-Paßwort gefragt. Anschließend wird das Verzeichnis in den CVS-Server importiert, was ggf. ein Weilchen dauern kann.

Jetzt legt man ein Backup des "alten" Codes an, den man gerade auf den CVS-Server kopiert hat. In Zukunft arbeitet man nicht mehr mit dieser Version, sondern mit einer frisch ausgecheckten CVS-Version. Die erhält man mit

cvs checkout Verzeichnisname

Dadurch bekommt man eine Arbeitskopie des Sources in für CVS verdaulicher Form. Man sollte auf keinen Fall vergessen, daß jede neue oder gelöschte Datei bzw. Verzeichnis CVS bekanntgegeben werden muß. Weitere Informationen dazu gibts im oben genannten Buch.

5. Webseiten für das Projekt auf den Webserver laden

Ich gehe davon aus, daß bereits ein paar Seiten zum Projekt existieren, die nun auch auf SourceForge zur Verfügung stehen sollen, damit man einen Eindruck von Ihrem Projekt gewinnen kann. Sollten Sie noch keine Seiten erstellt haben und auch nicht wissen, wie Sie das am besten anstellen, dann ist SelfHTML von Stefan Münz die richtige Adresse für Sie.

Um auf dem Projektaccount zu arbeiten, loggt man sich per SSH auf SourceForge ein, z.B. mit

ssh -l Username Projektname.sourceforge.net

Anschließend findet man sich im Home-Verzeichnis des Projektes wieder. Von hier aus werden alle Projektdateien im Verzeichnis /home/groups/Projektname abgelegt. Dort sind alle Webseiten im Verzeichnis ht_docs untergebracht. Bisher befindet sich in diesem Verzeichnis nur eine Datei index.php, die besagt, daß bisher noch keine Webseiten zur Verfügung stehen.

Die Webseiten werden mit dem Kommando scp übertragen, da&azlig; mit ssh ausgeliefert wird. Es wird folgendermaßen benutzt:

scp lokale_Dateien_zum_übertragen Username@shell.sourceforge.net:/home/groups/Projektname/ht_docs

Ich empfehle Ihnen, die Dateien mit gzip zu packen und mit scp zu übertragen. Loggen Sie sich anschlie&zlig;end mit ssh auf SourceForge ein und entpacken Sie das Archiv mit gunzip.

Sie fragen Sich vielleicht, was Sie auf Ihre Webseiten schreiben sollen. Neben einer Beschreibung des Projekt könnten Sie folgendes unterbringen:

  1. ein Link auf die SourceForge-Projektseite
  2. Links zu den Mailinglisten (die wir gleich anlegen)
  3. ein SourceForge-Icon und einen Zugriffszähler; Informationen dazu gibts bei der "Site Documentation" von SourceForge
  4. ein Link zum Webinterface des CVS-Servers unter http://cvs.sourceforge.net/cgi-bin/cvsweb.cgi/?cvsroot=Projektname
  5. einige für Ihr Projekt spezifische Informationen, wie CVS benutzt werden kann/soll

Das sind natürlich nur einige Anregungen. Lassen Sie Ihre Fantasie spielen und versuchen Sie sich vorzustellen, was Besucher Ihrer Seiten interessieren könnte.

6. Mailinglisten anlegen

Auch, wenn sich bei einem Projekt noch nicht viel tut oder nur ein einzelner Entwickler daran arbeitet, sollte man in die Zukunft denken. Ich empfehle, drei Standardlisten für jedes Projekt anzulegen. Eine devel-Liste für Entwickler, eine users-Liste für die Benutzer und eine announce-Liste, auf der neue Versionen angekündigt werden.

Die Listen können im Verwaltungsbereich des Projektes mit der Option "Mailing List Admin" angelegt werden. Das sollte keine Probleme bereiten, daher gehe ich darauf nicht weiter ein.

7. Mit CVS vertraut machen

Hier sind die wichtigsten Kommandos, die man eigentlich ständig braucht. Alles weitere muß man im schon genannten Handbuch nachschlagen.

Die lokale Arbeitskopie des Sources mit der aktuellen Version auf dem Server zu synchronisieren:

cvs -z3 update -Pd

(dazu muß man CVSROOT und CVS_RSH wie oben beschrieben eingestellt haben).

Die Änderungen an einer Datei auf den CVS-Server einspielen:

cvs commit -m "Beschreibung der Veränderungen." Dateiname

Eine neue Datei in das CVS repository aufnehmen:

cvs add Dateiname
cvs commit -m "Dateiname hinzugefügt" Dateiname

... und nun kanns losgehen!
